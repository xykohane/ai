逻辑拆解：
步骤 说明
allData = ref([...mockUserList]) 复制一份 Mock 数据，模拟"数据库"
filteredData（computed） 根据搜索关键词过滤，自动响应 keyword 变化
total（computed） 过滤后的总条数，驱动分页组件
list（computed） Array.slice() 切片实现前端分页
page = 1 在搜索时重置 避免搜索后停留在不存在的页码
splice 删除 直接操作数组，Vue 响应式自动更新表格
前端分页 vs 后端分页： 我们现在用的是前端分页——数据全在内存中，用 slice() 切片显示。等后端接口好
了（Day10+），会改为后端分页——每次只请求一页的数据，适合数据量大的场景。
computed 的好处： filteredData、total、list 都是 computed，当 keyword 或 page 变化时自动重
新计算，不需要手动调用 fetchList()。这是 Composition API 响应式系统的威力。

4、分页逻辑详解
分页是后台管理系统的标配功能。当前我们用前端分页（数据全在内存中用 computed + slice() 切片），后续对
接后端后会改为后端分页：
// 三个核心状态（v-model 双向绑定到 el-pagination）
const page = ref(1) // 当前页码（从 1 开始）
const pageSize = ref(10) // 每页显示几条
// computed 自动计算
const total = computed(() => filteredData.value.length) // 过滤后的总条数
const list = computed(() => {
const start = (page.value - 1) * pageSize.value
return filteredData.value.slice(start, start + pageSize.value)
})
el-pagination 组件属性：
属性 作用
v-model:current-page 双向绑定当前页码
v-model:page-size 双向绑定每页条数
:total 总记录数，用于计算总页数和页码按钮数量
:page-sizes="[10, 20, 50]" 每页条数的可选列表
layout="total, sizes, prev, pager, next" 分页组件的布局，显示哪些元素
@size-change 每页条数变化时触发的回调
@current-change 页码变化时触发的回调

=== Page 11 ===
前端分页 vs 后端分页： 我们用的是后端分页——前端只传 page 和 size，后端返回对应页的数据。另一种是
前端分页——一次性加载所有数据，前端用 Array.slice() 切片。数据量大时（几千条以上）必须用后端分
页，否则一次加载太慢。
