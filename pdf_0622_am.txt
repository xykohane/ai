1、Pinia 状态管理
Pinia 是 Vue3 官方推荐的状态管理库，用于管理跨组件共享的状态。在我们的项目中，"当前登录用户信息"需要在
AppLayout、Dashboard、User 等多个页面中使用，这就是典型的共享状态场景。
1.1 为什么需要状态管理
Day08 中我们用 localStorage 存 Token，但存在几个问题：
问题 localStorage 的局限 Pinia 的解决方案
用户信息存在哪？ 只存了 token 字符串，没有用户详情 Store 中存完整的 user 对象
多页面共享？ 每个页面都要 getItem 读取 任何组件直接 useUserStore().user
登录状态判断？ localStorage.getItem('token') userStore.isLoggedIn（computed）
退出登录？ 手动 removeItem 多个 key userStore.logout() 一行搞定
Pinia 的核心思想： 把需要共享的状态放在一个"全局仓库"（Store）中。任何组件都可以直接读写这个仓库，
不需要通过 props 层层传递。
npm install pinia

import { setToken, removeToken, getToken } from '../utils/token'
export const useUserStore = defineStore('user', () => {
// ---- State（状态）----
const user = ref(null) // 当前用户信息对象
const token = ref(getToken()) // JWT Token（从 localStorage 恢复）
// ---- Getters（计算属性）----
const isLoggedIn = computed(() => !!token.value)
const isAdmin = computed(() => user.value?.role === 'admin')
// ---- Actions（操作方法）----
async function login(data) {
// 1. 调用后端登录接口
const res = await loginApi(data)
// 2. 保存 Token
token.value = res.data.token
setToken(res.data.token)
// 3. 获取用户详细信息
await fetchUserInfo()
}
async function fetchUserInfo() {
const res = await getCurrentUser()
user.value = res.data
}
function logout() {
user.value = null
token.value = null
removeToken()
}
// 返回所有需要暴露的状态和方法
return { user, token, isLoggedIn, isAdmin, login, fetchUserInfo, logout }
})

Composition API 风格： defineStore('user', () => { ... }) 使用函数式写法，和 Vue3 组件的
<script setup> 风格一致。另一种选项式写法（defineStore('user', { state, getters, actions

1.3 在组件中使用 Store
<script setup>
import { useUserStore } from '../../stores/user'
const userStore = useUserStore()
// 读取 state
console.log(userStore.user) // 用户信息或 null
console.log(userStore.token) // Token 字符串
// 读取 getter
console.log(userStore.isLoggedIn) // true / false
console.log(userStore.isAdmin) // true / false
// 调用 action
await userStore.login({ username: 'admin', password: '123456' })
userStore.logout()
</script>
操作 代码 说明
读取 state userStore.user 直接访问，不需要 .value
读取 getter userStore.isLoggedIn 计算属性，自动缓存
调用 action userStore.login(data) 异步方法，返回 Promise
修改 state userStore.user = null 直接赋值，Pinia 自动追踪变化
Pinia vs ref 的区别： 在组件模板中直接用 {{ userStore.user }}，在 <script setup> 中也直接用
userStore.user（不是 userStore.user.value）。Pinia 内部对 state 做了自动解包处理。

2、Token 工具函数
Day08 中我们直接在登录页写 localStorage.setItem('token', ...)，现在把它封装为独立工具函数，方便统一
管理：
// utils/token.js
const TOKEN_KEY = 'ai_companion_token'
export function getToken() {
return localStorage.getItem(TOKEN_KEY)
}
export function setToken(token) {
localStorage.setItem(TOKEN_KEY, token)
}
export function removeToken() {
localStorage.removeItem(TOKEN_KEY)
}
为什么单独封装？ 如果直接在各处用 localStorage.getItem('ai_companion_token')，key 字符串散布在
多个文件中。一旦需要改 key 名称，要改很多地方。封装后 key 集中管理在一处，改一处即可。这是"常量提
取"的基本编码规范。

3.1 创建 Axios 实例
// api/request.js
import axios from 'axios'
import { ElMessage } from 'element-plus'
import { getToken } from '../utils/token'
const service = axios.create({
baseURL: import.meta.env.VITE_API_BASE_URL, // 从环境变量读取
timeout: 15000, // 请求超时 15 秒
headers: { 'Content-Type': 'application/json' },
})
配置项 含义
baseURL API 基地址，所有请求自动拼接前缀
timeout 请求超时时间，超过 15 秒自动取消
headers 默认请求头
环境变量： import.meta.env.VITE_API_BASE_URL 从 .env.development 文件读取。开发环境设为
http://localhost:8080，生产环境设为实际域名。Vite 约定只有 VITE_ 开头的环境变量才会暴露给前端代
码。
3.2 请求拦截器：Token 自动注入
// ---- 请求拦截器：自动注入 Bearer Token ----
service.interceptors.request.use(
(config) => {
const token = getToken()
if (token && config.headers) {
config.headers.Authorization = `Bearer ${token}`
}
return config
},
(error) => Promise.reject(error),
)

拦截器的好处： 如果没有拦截器，每个 API 调用都要手动加 Token：request.get('/api/users', {
headers: { Authorization: 'Bearer ' + token } })。有了拦截器，所有请求自动携带 Token，业务代
码完全不需要关心认证细节。
3.3 响应拦截器：统一解包与错误处理
// ---- 响应拦截器：解包 ApiResponse，统一处理错误 ----
service.interceptors.response.use(
(response) => {
const res = response.data
// 不是标准 ApiResponse 格式，直接返回
if (res.code === undefined) {
return response
}
// 成功：解包返回 data 字段
if (res.code === 200 || res.code === 0) {
return res.data
}
// 业务错误（code 非 200）
ElMessage.error(res.message || '请求失败')
return Promise.reject(new Error(res.message || '请求失败'))
},
(error) => {
const status = error.response?.status
if (status === 401) {
// Token 过期或无效
ElMessage.error('登录已过期，请重新登录')
localStorage.removeItem('ai_companion_token')
window.location.href = '/login'
} else if (status === 403) {
// 没有权限
ElMessage.error('没有权限访问该资源')
} else if (status === 500) {
// 服务器错误

=== Page 7 ===
ElMessage.error('服务器内部错误，请稍后再试')
} else {
// 其他错误
ElMessage.error(error.response?.data?.message || error.message || '网络错误')
}
return Promise.reject(error)
},
)
响应拦截器做了三件事：
情况 处理方式
成功（code 200 或 0） 解包 res.data，调用方直接拿到业务数据
业务错误（code 非 200） 弹出错误提示，reject 错误让调用方 catch
HTTP 错误（401/403/500） 弹出对应提示，401 自动清除 Token 并跳转登录页
解包的好处： 后端统一返回 { code: 200, message: "ok", data: { ... } } 格式。拦截器解包后，调用
方直接拿到 data，不用每次都写 res.data.data。这和 HarmonyOS 中 HttpClient 的拦截器思路完全一
致。
401 为什么用 window.location.href 而不是 router.push？ 因为 router 实例不在 request.js 中。用
window.location.href 会刷新整个页面，确保所有状态（包括 Pinia store）都被彻底重置，避免残留的脏数
据。

return request.get('/api/skills/tree')
}
export function createSkill(data) {
return request.post('/api/skills', data)
}
export function updateSkill(id, data) {
return request.put(`/api/skills/${id}`, data)
}
export function deleteSkill(id) {
return request.delete(`/api/skills/${id}`)
}
模块化的好处： 每个 API 文件只包含一类业务的接口，文件小、职责单一。后端接口路径变了，只需要改对应
的 API 文件，不影响页面代码。这是"单一职责原则"的体现。

│ → Pinia state 清空（内存中的数据丢失）
│ → token = getToken()（从 localStorage 恢复 Token）
│ → AppLayout 的 onMounted 检测到"已登录但没有用户信息"
│ → 自动调用 fetchUserInfo() 恢复用户信息
│ → 页面恢复正常显示
为什么登录成功后还要调 fetchUserInfo()？ 因为登录接口只返回 Token 和基本信息（userId、
username），不返回完整的用户信息（角色、头像、昵称等）。需要拿着 Token 再请求一次 /api/users/me
获取完整数据。
